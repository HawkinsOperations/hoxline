# ProofOps Doctrine

Doctrine: AI is not the authority. Evidence is.

Hoxline exists because AI-assisted security work needs a control plane that can separate useful assistance from unsupported claims.

## Control Principle

AI-assisted work can propose artifacts, wording, checks, summaries, and next actions. It cannot authorize a claim. A claim can only move forward when the evidence graph, promotion state, review gate, ProofCard, and Claim Authority agree that the claim is supported.

## Core Loop

AI-assisted security work
→ Artifact Intake
→ Evidence Graph
→ Telemetry Contract Check
→ Controlled Validation
→ Runtime Candidate Ledger
→ Signal Observation
→ Human Review Gate
→ ProofCard
→ Claim Authority
→ Safe Claim / Blocked Claim

## Doctrine Rules

* Evidence outranks generated text.
* Missing evidence blocks or constrains claims.
* Runtime candidate state is not observation.
* Signal observation must be explicit and traceable.
* Human review is a gate in the loop, not a substitute for evidence.
* ProofCards summarize evidence state; they do not manufacture evidence.
* Claim Authority governs claim wording and final disposition.
* Claim Firewall is one internal enforcement capability inside Claim Authority.

## Product Ceiling

This doctrine file defines product rules only. It does not assert proof of runtime behavior, signal observation, external safety status, final authorization, or deployment status.
